**訂正**: Next.js 16で、Express.jsのミドルウェアとの混同を避ける目的で、公式に`middleware.ts` → `proxy.ts`への名称変更が行われている（`next@16.2.7`はこの仕様に対応済みバージョン）。したがって`proxy.ts`のままで正しく、リネームは不要だった。

**実装上の実際のバグ**: ファイル名ではなく、ロジック側に問題があった。

- 当初のコード: `PROTECTED_PATHS = ["/profile", "/matching", "/chat"]`という「保護パス方式」で、`/`や`/login`が対象外だったため、未ログインでもトップページに入れてしまっていた。
- 1回目の修正で`PUBLIC_PATHS`という名前に変えたが中身は保護パスのままで、`/login`自体がログイン必須ページ扱いになり、無限リダイレクトループ（`ERR_TOO_MANY_REDIRECTS`）が発生した。
- 最終的に、`PUBLIC_PATHS`の中身を実際に「ログイン不要な公開ページ（`/login`・`/register`）」に修正し、それ以外は全てログイン必須という設計に変更して解決。

**最終コード**（`src/proxy.ts`）:
```typescript
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

// ログイン不要でアクセスできるパス
const PUBLIC_PATHS = ["/login", "/register"];

export default function proxy(request: NextRequest) {
  const token = request.cookies.get("access_token");
  const { pathname } = request.nextUrl;

  const isPublic = PUBLIC_PATHS.some((path) => pathname.startsWith(path));

  if (!token && !isPublic) {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  if (token && isPublic) {
    return NextResponse.redirect(new URL("/", request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/((?!_next/static|_next/image|favicon.ico).*)"],
};
```

**動作確認済み**: 未ログインで`/`や`/login`以外にアクセス→`/login`へリダイレクト。ログイン済みで`/login`・`/register`にアクセス→`/`へリダイレクト。無限ループは解消。


### ✅ #5 実施内容: 物理削除 → 論理削除（is_active切り替え）方式への変更

**背景**: 削除ボタン押下時に`Failed to fetch`エラーが発生。実体はバックエンドの500エラー（FastAPI/StarletteはCORSMiddleware内側で未処理例外が起きるとCORSヘッダーを付与できず、ブラウザ側では「CORSブロック」に見えていた）。真因は、削除対象ユーザーが`likes`/`messages`テーブルで他ユーザーと紐付いており、`ON DELETE CASCADE`未設定のため`db.delete(db_user)`時に外部キー制約違反（IntegrityError）が発生していたこと。

**対応方針**: 物理削除ではなく`is_active`フラグを`False`にする論理削除方式に変更（FK制約の問題を根本的に回避）。

**Backend**

`crud/user.py`: 一覧・詳細取得のフィルタ方針を整理し、無効化・有効化の関数を追加。
```python
def get_users(db: Session, skip: int = 0, limit: int = 100):
    """ユーザー一覧取得（有効なユーザーのみ）"""
    return db.query(User).filter(User.is_active == True).offset(skip).limit(limit).all()

def get_user(db: Session, user_id: int):
    """ユーザー1件取得（無効化されていてもアクセス可・本人の再有効化用）"""
    return db.query(User).filter(User.id == user_id).first()

def get_user_by_email(db: Session, email: str):
    """メールアドレスでユーザーを検索（無効化されていてもログイン可能にする）"""
    return db.query(User).filter(User.email == email).first()

def deactivate_user(db: Session, user_id: int):
    """ユーザーを無効化（論理削除）"""
    db_user = db.query(User).filter(User.id == user_id).first()
    if db_user:
        db_user.is_active = False
        db.commit()
        db.refresh(db_user)
    return db_user

def activate_user(db: Session, user_id: int):
    """ユーザーを有効化"""
    db_user = db.query(User).filter(User.id == user_id).first()
    if db_user:
        db_user.is_active = True
        db.commit()
        db.refresh(db_user)
    return db_user
```

> 設計判断: `get_users`（一覧）のみ`is_active`でフィルタし、`get_user`（詳細）・`get_user_by_email`（ログイン認証）はフィルタしない方針にした。理由は、無効化されたユーザーもログイン・自分のプロフィールへのアクセスはできるようにし、自分で再有効化できる導線を残すため。

##  1 | `create_user`でプロフィール項目（`phone`/`penname`/`introduction`/`personality`/`work`）が`"テスト"`固定でハードコードされている | `crud/user.py`
- 修正内容
 - 空欄（None）にするのが妥当です。理由と修正案を説明します。
なぜ空欄がいいか
登録時（UserCreate）に必要なのは本来name・email・passwordだけです。penname（ペンネーム）・introduction（自己紹介）・personality（性格）・work（作品URL）は、新規登録直後にはまだ本人が入力していない情報なので、"テスト"のような固定文字列を入れるのではなく、未入力状態（NULL）で登録し、後からプロフィール編集画面（ProfileEditForm）で埋めてもらう、という流れが自然です。

修正コード
```
def create_user(db: Session, user: UserCreate):
    """ユーザー作成"""
    db_user = User(
        name=user.name,
        email=user.email,
        hashed_password=hash_password(user.password),
    )
    db.add(db_user)
    db.commit()
    db.refresh(db_user)
    return db_user
```
"テスト"固定の項目を単純に削除するだけでOKです。models/user.py側を見ると、これらのカラムは以下のようにすでにnullable扱いになっています（nullable=Trueが明示されていないものも、String型でデフォルト制約がないので実質NULL可）。
pythonpenname = Column(String(255), nullable=True)
introduction = Column(String(1000))   # nullable指定なし → デフォルトでNULL許可
personality = Column(String(255))     # 同上
work = Column(String)                 # 同上
なのでモデル側の変更は不要で、crud/user.pyのcreate_userだけ直せば大丈夫です。
